# ai-pipes: AI-Generated DeFi Indexers

**Date:** 2026-03-18
**Repo:** `karelxfi/ai-pipes`
**Status:** Approved design

## Overview

A public GitHub repo showcasing AI-generated blockchain indexers built with SQD's Pipes SDK. Each indexer targets a top DeFi protocol (sourced from DeFiLlama), is fully generated by Claude Code using the `subsquid-labs/agent-skills` skill set, and runs locally out of the box.

Each example tells a compelling data story вЂ” not generic event indexing, but protocol-specific insights (liquidation cascades, MEV analysis, yield curve dynamics, whale tracking). Every example includes a shareable dashboard with charts designed to look great as screenshots on X.

The repo serves as both a showcase and a reproducibility archive вЂ” every example includes the exact prompt, SDK versions, and generation metadata so anyone can recreate or extend it.

## Goals

1. Generate working Pipes SDK indexers for the top 100 DeFi protocols by TVL
2. Each example has a unique, compelling angle specific to the protocol's mechanics
3. Each example includes data validation and a shareable visual dashboard
4. Make every example reproducible: same prompt + same versions = same result
5. Commit the Claude Code configuration publicly for full transparency
6. Organize by VM type (EVM, Solana, Hyperliquid) for easy browsing

## Compelling Examples: The Angle

Every example must have a unique, protocol-specific angle that makes it interesting to share. Generic "index Transfer events" is not acceptable.

### Angle Selection Principles

1. **Protocol-specific mechanics** вЂ” leverage what makes each protocol unique
2. **Data story** вЂ” the indexer should answer a question or reveal a pattern
3. **Visual impact** вЂ” the dashboard chart should tell the story at a glance

### Example Angles (illustrative, not exhaustive)

| Protocol | Bad (generic) | Good (compelling) |
|----------|--------------|-------------------|
| Aave V3 | Index supply/borrow events | Track liquidation cascades вЂ” when one liquidation triggers others |
| Uniswap V3 | Index Swap events | Impermanent loss tracker вЂ” real-time IL for concentrated liquidity positions |
| Curve | Index pool events | Depeg detector вЂ” monitor stablecoin deviations from peg across pools |
| Pendle | Index yield events | Yield curve dynamics вЂ” implied vs realized yield over time |
| Lido | Index staking events | Validator performance вЂ” stETH discount/premium tracking |
| Morpho | Index lending events | Utilization rate heatmap вЂ” which markets are stressed |
| Compound V3 | Index supply events | Interest rate regime changes вЂ” detect rate model shifts |
| Kamino (Solana) | Index vault events | Auto-rebalancing efficiency вЂ” how well vaults track their ranges |
| Hyperliquid | Index fills | Whale tracker вЂ” large position opens/closes and their market impact |

The exact angle for each protocol is determined during generation based on what events/data are available and what makes the most interesting visual.

## Dashboard & Visualization

### Per-Example Dashboard

Each example includes a `dashboard/index.html` вЂ” a standalone HTML file that:

1. Connects to the local ClickHouse instance
2. Renders charts using [Lightweight Charts (TradingView)](https://github.com/nicehash/lightweight-charts) or similar
3. Shows the protocol-specific data story with 2-4 charts
4. Is styled to look good as a screenshot for sharing on X
5. Includes the protocol name, data description, and ai-pipes branding

### Design Requirements

- **Dark theme** вЂ” looks better in screenshots and on X
- **Self-contained** вЂ” single HTML file, CDN dependencies, no build step
- **Responsive** вЂ” works in browser, but optimized for 1200x675 screenshot ratio (X card size)
- **Branded** вЂ” subtle "Built with ai-pipes + SQD Pipes SDK" watermark

### Shared Template

`shared/dashboard-template/` provides a base HTML template with:
- TradingView Lightweight Charts loaded from CDN
- Dark theme CSS
- ClickHouse HTTP query helper
- Standard layout (title, subtitle, chart grid, watermark)

Each example's `dashboard/index.html` extends this pattern but is fully standalone (no imports from shared вЂ” the template is a starting point, not a runtime dependency).

## Data Validation

### validate.ts

Each example includes a `validate.ts` script that:

1. Connects to ClickHouse after the indexer has synced
2. Runs assertion queries to verify the data is correct:
   - Row count > 0 (data was actually indexed)
   - Schema matches expectations (correct columns, types)
   - Spot checks: known transactions/events appear in the data
   - Range checks: values are within expected bounds (no negative balances, timestamps in range)
3. Outputs pass/fail with details
4. Run via `npx tsx validate.ts`

### Validation in the Generation Workflow

The `/generate-indexer` command must run validation before marking an example as done. An example with failing validation stays in `failed` status.

## Protocol Selection

### Source

Fetch from `https://api.llama.fi/protocols`, sorted by TVL descending.

### Filtering Criteria

**Include:** DEXes, Lending, CDPs, Yield, Derivatives, Liquid Staking protocols with indexable smart contract events.

**Exclude:** CEXes (Binance, OKX, Bybit, etc.), canonical bridges with minimal contract logic, RWA tokens without on-chain lending/trading, staking pools with no contract events.

### Expected Breakdown

- ~50-60 EVM protocols
- ~10-15 Solana protocols
- ~3-5 Hyperliquid protocols
- For multi-chain EVM protocols, default to Ethereum mainnet unless the protocol is primarily deployed elsewhere

## Per-Example Structure

### META.json

```json
{
  "date": "2026-03-18",
  "pipes_sdk_version": "0.x.x",
  "agent_skills_version": "commit-sha-or-tag",
  "claude_model": "claude-opus-4-6",
  "prompt_file": "PROMPT.md",
  "angle": "Liquidation cascades вЂ” tracking when one liquidation triggers others",
  "runtime_status": "working",
  "validation_status": "passed"
}
```

- `runtime_status`: `working` | `broken` вЂ” does the indexer start and sync?
- `validation_status`: `passed` | `failed` | `untested` вЂ” does the data check out?

### PROMPT.md

The exact verbatim prompt used to generate the indexer. Nothing more.

### OUTPUT.md

Brief narrative of the generation: what happened, key decisions made, issues encountered and how they were resolved. Not a full transcript.

### README.md (per example)

- Protocol name and the specific angle/data story
- **Dashboard screenshot embedded at the top** вЂ” `dashboard/screenshot.png` committed to the repo and displayed via `![Dashboard](dashboard/screenshot.png)` so it's visible on GitHub without running anything
- How to run: `docker compose up -d && npm start`
- How to validate: `npx tsx validate.ts`
- How to view dashboard: open `dashboard/index.html` in browser
- At least one sample ClickHouse query and its expected output shape

### dashboard/screenshot.png

A screenshot of the rendered dashboard, captured after the indexer has synced and the dashboard is populated with real data. This is the shareable visual вЂ” it should look compelling enough to post on X directly from the README.

### CLAUDE.md

Root instructions for the agent:

1. **Setup:** Install agent-skills via `npx skills add subsquid-labs/agent-skills --all`
2. **Workflow:** When generating an indexer:
   - Use the `pipes-new-indexer` skill
   - Choose a compelling, protocol-specific angle (not generic event indexing)
   - Pin the Pipes SDK version in `package.json`
   - Generate `META.json` with date + all versions
   - Save the initial prompt to `PROMPT.md`
   - Document the generation in `OUTPUT.md`
   - Create `validate.ts` with data assertions
   - Create `dashboard/index.html` with TradingView charts telling the data story
   - Verify the example runs locally before marking done
   - Run validation before marking done
3. **Conventions:** Follow agent-skills CLAUDE.md (Rule 0: read docs first)
4. **Tracking:** Update `protocols.json` status after each generation
5. **Agent-skills version:** Record the installed commit SHA in META.json. Note: `npx skills add` installs latest; exact reproducibility depends primarily on Pipes SDK version pinning.

### .claude/commands/generate-indexer.md

Slash command `/generate-indexer` that:

1. Asks for protocol name (or picks next undone from `protocols.json`)
2. Researches the protocol to find a compelling angle
3. Looks up protocol contracts/events
4. Invokes `pipes-new-indexer` skill
5. Scaffolds the example directory under the correct VM folder
6. Generates all metadata files
7. Creates the dashboard
8. Creates validation script
9. Tests locally: indexer runs, validation passes, dashboard renders
10. Captures `dashboard/screenshot.png` from the rendered dashboard
11. Updates `protocols.json` status

- Each example is a self-contained commit
- `protocols.json` is the source of truth for progress
- The `/generate-indexer` command handles the full workflow
